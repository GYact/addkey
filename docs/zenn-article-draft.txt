## TL;DR

- AIエージェントとのコーディング中、つい **APIキーをチャットに貼ってしまう** 事故を防ぐ CLI を作りました。
- `addkey OPENAI_API_KEY` と打つと **入力非表示のダイアログ** が出て、値は標準出力にもシェル履歴にもAIの会話ログにも残らず `.env` に保存されます。
- SOPS + age と組み合わせ「**git上は暗号化・ディスク上は平文**」を実現。アプリは普通の `.env` をそのまま読めます。
- Apache-2.0 で OSS 公開: https://github.com/GYact/addkey

## 背景: 「キーを貼る」問題

LLM コーディングが当たり前になって、こんな場面が増えました。

> 「この API 叩くコード書いて。キーはこれ → `sk-...`」

一度チャットに貼ったシークレットは、会話履歴・ベンダー側ログ・ローカルのセッション
ファイルなど、自分が把握しきれない場所に残り得ます。`.env` をエディタで開いて
AIに見せる、ターミナルに `export` して履歴に残す、といった経路も同じです。

「気をつける」では再発します。**そもそも貼る導線をなくす** のが本質的な解決です。

## 設計方針

3つだけ守りました。

1. **値を絶対に標準出力/標準エラーに出さない** — エコーした時点で履歴・ログに残るため。
2. **平文 `.env` はディスクに残す** — アプリもAIエージェントも従来どおり動く(ゼロ摩擦)。
3. **コミットする正本は暗号化版 `.env.enc`** — SOPS + age で値だけ暗号化、キー名は平文なので差分レビューが効く。

## 使い方

```bash
brew install sops age
git clone https://github.com/GYact/addkey.git && cd addkey
./install.sh
addkey init           # age鍵と公開鍵を用意(初回のみ)
```

あとはプロジェクトで:

```console
$ addkey OPENAI_API_KEY
# GUIダイアログが出る → キーを入力 → 値は一切表示されない
/path/to/project/.env: 'OPENAI_API_KEY' added (.enc synced) (value hidden).
```

`.env` には平文、`.env.enc` には暗号化版が同期されます。`.env.enc` だけをコミット。

```text
# .env.enc (コミット可: 値は暗号化、キー名は読める)
OPENAI_API_KEY=ENC[AES256_GCM,data:...,iv:...,tag:...,type:str]
```

新しいマシンや復旧時は `senv-pull` で `.env.enc` → `.env` を復号、平文を編集したら
`senv-push` で再同期、中身の確認は `senv-cat .env.enc`、暗号化したまま編集は
`senv-edit .env.enc` です。

## 実装メモ

### 値を出さない入力

macOS では osascript の `display dialog ... with hidden answer` を使い、入力値は
シェル変数にだけ保持します。Linux は zenity、無ければターミナルの `read -s` に
フォールバック。テスト/CI 用に環境変数 `ADDKEY_VALUE` でダイアログを省略できます。

```bash
read -rs _out </dev/tty   # 端末から直接読む。プロンプトは stderr へ
```

### 公開鍵をハードコードしない(ここ重要)

OSS 化で一番やってはいけないのが **作者の age 公開鍵をスクリプトに埋め込むこと** です。
そのまま配ると全ユーザーが作者宛に暗号化してしまい、本人が復号できなくなります。
そこで recipient(公開鍵)は次の順で解決します。

1. 環境変数 `SOPS_AGE_RECIPIENT`
2. `~/.config/senv/recipient`(`addkey init` が書き込む)
3. 秘密鍵から `age-keygen -y` で自動導出

これで「自分の鍵で暗号化 → 自分で復号」がゼロ設定で成立します。

### git からの平文の追跡解除

`sopsify` は暗号化版を作ると同時に、追跡済みの平文を `git rm --cached` します。
ただし **履歴は書き換えられない** ので、既に push 済みなら鍵のローテーションが必要、
とドキュメントに明記しています。

## 落とし穴

- `senv-cat` は仕様上、復号した値を表示します。ログに残る場所で使わないこと。
- age **秘密鍵** はパスワード同等。コミット厳禁・要バックアップ。
- 一度漏れたキーは取り消せません。ツールは再発防止であって、流出後の救済ではない。

## おわりに

「便利さ」と「秘密を漏らさない」を両立させるには、人間の注意力に頼らず
**導線そのものを設計し直す** のが効きます。よかったら使って・Star・Issue ください。
